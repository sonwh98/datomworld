# `dao.space` + `dao.flow` vs. MPI: Fills the gap, and goes beyond

Date: 2026-07-09
Evaluating the claim: *"`docs/design/dao.space.md` with `docs/design/dao.flow.md` fills the gap with MPI and goes beyond it."*

**Terminology.** `dao.space` is the tuple space, the associative coordination surface agents match over; it is *not* the storage. Durable storage is `dao.jing`, the content-addressed store of opaque bytes built on `dao.stream`, the append-only log. When this document speaks of durability, immutability, replay, or history, it means `dao.jing` underneath `dao.space`; when it speaks of matching, coordination, and the collective vocabulary, it means `dao.space` itself. The protocol layer is `dao.flow`. The claim under review is really about all three together.

**Implementation status.** This document evaluates the architecture *as designed*, not the build as it stands today. Several capabilities discussed here are specified but not yet implemented, and the source design docs mark them explicitly; where a claim depends on unbuilt machinery, this document flags it inline rather than presenting target surface as accomplished fact.

## What is MPI and its Use Cases?

**MPI (Message Passing Interface)** is the dominant standard for distributed memory parallel computing. Created in the 1990s, it provides a standardized API for processes to communicate via messages. 

**Core MPI Use Cases:**
- **High-Performance Computing (HPC) & Supercomputing**: Weather forecasting, fluid dynamics, quantum mechanics, and molecular dynamics.
- **Tightly-Coupled Parallel Algorithms**: Algorithms requiring frequent, low-latency synchronization (e.g., matrix multiplication, fast Fourier transforms).
- **Static, Homogeneous Clusters**: Environments where a fixed number of identical nodes execute the same program (SPMD - Single Program, Multiple Data) and address each other by fixed numerical ranks (e.g., Rank 0, Rank 1).

MPI excels at moving raw buffers of data between a known set of active participants with microsecond latency using primitives like Point-to-Point (Send/Recv) and Collectives (Broadcast, Scatter/Gather, Reduce, Barrier).

## How datom.world Handles MPI Use Cases

Every MPI primitive maps onto a `dao.flow` convention within the `datom.world` tuple space. The collective vocabulary is recovered as stigmergic patterns:

- **"Broadcast" is write-availability, not synchronous delivery.** A typed datom appended to the shared medium becomes matchable by every agent that matches its type; no agent is addressed, and none is pushed to. This is Linda generative broadcast (anyone may read, asynchronously, on their own poll), *not* `MPI_Bcast` (synchronous delivery to a fixed, known rank set at a barrier, with all participants blocked until receipt). The append-only log plus global associative match give you the read-availability half for free; the synchronous-delivery half is a different thing, recovered (if at all) by a barrier convention, and it is the subject of the ordering and fail-fast caveats later in this document.
- **Reduce / Allreduce = join agent + Datalog.** Distributed gradient aggregation (as in ML training) has gradient result datoms from different nodes written to `dao.space`; a join agent Datalog-queries all gradient results and folds them, and allreduce writes the folded result back as a type others match. This works today for commutative folds (gradient sums) on the joins-only Datalog that is implemented. It depends on two pieces of query machinery that are target surface: *current-state masking* (treating the accreted log, retractions and all, as a clean result) and, for the completeness check that bounds the fold, the membership logic discussed below. See `docs/design/dao.space.md`, "The Query Library," for the joins-only status.
- **Barrier / scatter / gather / all-to-all** all compose from the same two moves (append a typed group, match/join on type). A barrier is not a blocking wall; it is a coordinator matching N `:arrive` datoms and emitting a `:release` datom. The hard part is not the pattern but knowing *N*: who counts as a participant, and when the coordinator may conclude everyone who matters has arrived. That is the membership / completeness problem, the recurring caveat of this document and the load-bearing unimplemented piece (see below).
- **The communicator is replaced by content-defined groups.** Where MPI fixes a ranked communicator, `dao.flow`'s "group" is *the set of datoms carrying a given type value plus the agents matching it*, dynamic, content-defined, with no ranks. And `dao.space` is explicit that "membership is intake, not identity": streams join and leave, the medium persists. Associative matching by content works today; canonical content-addressed *identity* (a hash-derived, cross-stream stable id) is not yet load-bearing (ADR 0001, Open Question 3), so "content-addressed" here means *matched by content*, not yet *identified by content hash*.
- **Point-to-point is deliberately dropped**, not lost. Directed messaging is recovered by convention (a type only one agent matches), or falls back to `dao.stream.rpc` when strict request/response is needed. The thesis is that generative communication usually subsumes the need for it.

## Cases Where MPI is More Appropriate

Despite the expressive equivalence, MPI remains the superior choice for specific workloads, particularly where "goes beyond" is still aspirational for `datom.world`:

1. **Microsecond-Latency Synchronous Kernels**: For tight-loop, barrier-synchronized numerical computing (the MPI heartland), the overhead of datom creation, appending to a log, indexing (B-Trees), and Datalog matching would absolutely dominate execution time. MPI moves raw memory buffers directly; it is topology-optimized for million-core fabrics. "Fills the gap" is true for coordination semantics, not for this performance niche.
2. **Fixed-Topology, Known-Membership Algorithms**: The hard part of collectives in `datom.world` is delegated to a scheduler agent, and it is a known hard problem in disguise. Deciding "all N participants have contributed" (the termination condition for a reduce or barrier) is *distributed termination detection*; over MPI's fixed communicator it is free, but over `dao.space`'s open, intake-based membership it is provably harder, because "who counts as a participant" is dynamic and content-defined rather than enumerated at startup. MPI gives you known membership for free via Communicators; `dao.space` must reconstruct an approximation of it. 
3. **Global Ordering Guarantees**: MPI collectives give strict ordering within the communicator for free. `dao.space` has per-stream order, but cross-stream ordering (ADR 0001, Open Question 2) is harder. For non-commutative reductions, MPI's guaranteed ordering is vastly simpler to reason about; this is the second load-bearing dependency (after membership) for any claim of replacing ordered MPI collectives.
4. **Fail-Fast Workloads**: MPI fails fast (a dead rank aborts the entire job). While `datom.world` is crash-only (readers block, writers resume), a tightly-coupled mathematical simulation often *prefers* to abort immediately if a single node dies, rather than waiting indefinitely for a datom that will never arrive. How a scheduler agent ejects a dead participant is currently unspecified.

## Cases Where MPI is Weak but datom.world is Strong

Assuming a complete `datom.world` system, it dramatically outperforms MPI in environments that are open, dynamic, heterogeneous, and require persistence:

1. **Dynamic Reconfiguration & Elasticity**: MPI assumes a fixed pool of ranks at startup (SPMD). In `datom.world`, agents (streams) can join and leave dynamically. The workflow topology is emergent based on the datoms flowing through the space, not a design-time graph. Agents redirect flow simply by writing new type values.
2. **Temporal Decoupling (Time-Shifted Execution)**: MPI's send/recv requires both the sender and receiver to be coupled in both identity and time. In `datom.world`, a producer appends a datom and dies; a consumer wakes up later and matches it.
3. **Partial Failure & Resilience**: MPI jobs shatter if one node dies. `datom.world` resolves concurrent claims on the read side as an event-sourcing rule, not a destructive update. Two workers claim the same task → both claims are recorded as datoms → a downstream reader sorts by timestamp and breaks ties by worker id, deterministically yielding one winner; exclusion is a query rule in the interpreters, not a guarantee the store enforces. (CRDTs appear elsewhere in the design, for *index merge* across streams, a separate concern; the claim-resolution mechanism itself is a deterministic query rule, not a CRDT.) MPI punts concurrent claims to the application.
4. **Auditability and Time-Travel**: MPI messages exist in flight and vanish. `dao.jing` preserves an immutable, queryable history of every claim, allocation, and decision. Resuming from a checkpoint is just replaying to a known cursor, with no special checkpoint infrastructure needed. 

## What datom.world Can Do That MPI Can't (Or is Impossible)

1. **Associative, Content-Addressed Coordination**: 
   MPI addresses by rank (e.g., "Send to Node 4"). `dao.flow` addresses by content. An agent writing `:ml/kernel-effect` does not know or care which kernel agent picks it up; strangers coordinate without being introduced. For open, multi-agent systems, this is a superpower MPI structurally lacks. Associative *matching* by content is implemented today; canonical content-addressed *identity* (hash-derived, cross-stream stable ids) is target surface, not yet load-bearing (ADR 0001, Open Question 3).
2. **The Interpreter Family (Moduli Space)**:
   MPI moves buffers. `dao.space` provides multiple interpretation surfaces over the exact same facts. You can run a Datalog relational query, a Graph traversal, an entity document fetch, or a Vector search (`dao.qi`) over the same canonical datom history. This is a database-theoretic idea with no MPI analog.
3. **In-Process Shared Storage Across Languages (JVM/JS/Dart/WASM)**:
   MPI is fundamentally transport-bound. `datom.world` scales from identical network semantics down to in-process sharing across heterogeneous runtimes using the same unified model. The "zero-copy" claim is scoped and worth stating precisely: what every runtime shares is the *`dao.jing` segment bytes* (immutable blobs each runtime reads lazily into its own datom representation, with no serialization round-trip for coordination state), not in-memory objects and not the bulk numerical payloads, which live in VRAM as handles outside `dao.space`. So this does not contest MPI's microsecond-kernel advantage (point 1 above, about moving raw buffers); it is a statement about coordination metadata spanning language boundaries cheaply, which MPI has no analog for at all.
4. **Governed-Computation Security & Provenance**:
   MPI assumes total trust within the cluster. `datom.world`'s architecture is designed for capability tokens, confined interpreters, and a provenance slot (`m`) on every datom, so that in target form you know *who* asserted *what* and *why*, making it safe for untrusted, multi-tenant agent collaboration. Implementation status matters here: the `m` slot exists structurally today but is defaulted rather than populated with rich provenance, and capability tokens plus the confined-interpreter / governed-computation mode are *specified but not yet implemented* (`docs/design/dao.space.md`, "Public mode only, today"; ADR 0002). This is an architectural advantage MPI lacks, not yet a deployed one.

## Punchline

The claim is correct as stated for the **coordination model** and the **durability/decoupling advances**. `dao.space` + `dao.flow`, over `dao.jing` storage, is Linda's generative communication plus Datomic's durable history, fused: it recovers MPI's collective vocabulary in a more decoupled, replayable, dynamically-reconfigurable form, and adds associative addressing, emergent topology, the interpreter family, shared in-process storage across languages, and governed-computation security that MPI structurally lacks.

Two honest qualifications on "goes beyond." First, much of the *beyond* is architectural rather than built: capability tokens and the confined-interpreter mode are specified but unimplemented, canonical content-addressed identity is not yet load-bearing, and the Datalog engine is joins-only. Second, "fills the gap and goes beyond" is true; "replaces MPI at what MPI is best at" is a stronger claim that remains to be earned. It depends on two pieces of machinery that are currently unspecified or deferred: the scheduler agent's **membership and completeness logic** (termination detection over an open, content-defined group), and **cross-stream ordering** for non-commutative, ordered collectives (ADR 0001, Open Question 2). Until those exist, the MPI heartland of microsecond synchronous kernels over known membership stays MPI's.
